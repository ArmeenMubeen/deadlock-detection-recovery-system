⚠ What is a deadlock?
A deadlock occurs when a set of processes are permanently blocked — each waiting for a resource held by another process in the group. No process can proceed without external OS intervention. All four Coffman conditions must hold simultaneously:
C1
Mutual exclusion
Only one process can hold a resource at a time
C2
Hold and wait
A process holds resources while requesting more
C3
No preemption
Resources cannot be forcibly taken away
C4
Circular wait
P1→P2→P3→P1 — a cycle of waiting
📊 Resource Allocation Graph (RAG)
PProcess nodes — circles. Represent running or waiting processes.
RResource nodes — squares with dots showing available instances.

▬ Assignment edge (R→P) — resource is currently held by process
- - Request edge (P→R) — process is blocked waiting for resource

Single-instance: cycle in RAG = deadlock (necessary & sufficient)
Multi-instance: cycle necessary but not sufficient — use Banker's
🔍 Detection algorithms
DFS cycle detection
  • Single-instance
  • DFS on RAG
  • Back-edge = cycle
  • O(V+E) time
  • Used in RAG tab
Wait-for graph
  • Remove resource nodes
  • Direct P→P edges
  • Same DFS cycle check
  • Faster, simpler
  • Single-instance only
Banker's algorithm
  • Multi-instance
  • Need≤Work check
  • O(n²×m) time
  • Avoidance strategy
  • Used in Banker's tab
🔧 Recovery & victim selection
Termination — Kill process(es) in the deadlock cycle. Simple but work is lost. Can terminate all at once or one-by-one with re-detection.

Preemption — Forcibly take a resource from a process. Process pauses and is re-queued. Risk of starvation if same process always chosen.

Rollback — Revert process to a saved checkpoint. Least data loss. Requires OS to maintain checkpoints periodically.

Victim score = (resources held × 2) + pending requests − process priority. Lowest score = selected as victim (cheapest to disrupt).
❓ How to Use DeadlockOS
This is a complete step-by-step guide for anyone new to this system — even if you have never studied deadlocks before. Read through all the steps in order for the best experience.
🔴 What is a Deadlock? (Start Here)
Imagine 4 cars at a 4-way intersection — each car is waiting for the car in front to move, but no car can move because they are all blocking each other. That is a deadlock in a computer system.

In an OS, Processes (running programs) need Resources (CPU, memory, files, printers) to work. A deadlock happens when:
P1 holds R1 and needs R2
P2 holds R2 and needs R1
→ Neither process can ever proceed. The system is stuck forever. This is a deadlock.
STEP 1 Load a Scenario or Build Your Own Graph
Option A — Quick Start (Recommended for beginners):
Look at the left panel → Quick scenarios. Click one of these chips:
3-process deadlock — Best for beginners. Loads 3 processes + 3 resources in a classic circular deadlock (P1→R2→P2→R3→P3→R1→P1).
Dining philosophers — 5 processes (philosophers) and 5 resources (forks) — a famous deadlock example from OS theory.
Safe state — Loads a scenario with NO deadlock, so you can see what a safe system looks like.
Option B — Build manually:
In the left panel, use Add process and Add resource to create nodes. Then add edges by:
• Clicking Click nodes on canvas → click a source node → click a target node → choose edge type in the popup, OR
• Using the Manual entry boxes: type node names (e.g. R1, P1) and click Add edge.
STEP 2 Understand the Graph (RAG Visualizer Tab)
The canvas shows the Resource Allocation Graph (RAG). Here is how to read it:
P1
Circle = Process
A running program that needs resources
R1
Square = Resource
A system resource (CPU, file, printer)
R ——→ P
Solid line = Assignment
Resource is currently HELD by process
P - - → R
Dashed line = Request
Process is WAITING for the resource
Red nodes = nodes involved in the deadlock cycle (appears after detection)
STEP 3 Run Deadlock Detection
In the RAG Visualizer tab, click the ▶ Run detection button at the top of the canvas.

The system will run DFS (Depth-First Search) on the graph to look for cycles.

🔴 If DEADLOCK FOUND: A red banner appears at the bottom of the canvas. Deadlocked nodes turn red. The detection log on the right shows the full cycle chain (e.g. P1 → R2 → P2 → R3 → P3 → R1 → P1). The system automatically runs Banker's Algorithm and shows the result.
✓ If NO DEADLOCK: A green banner appears confirming the system is in a safe state. No nodes turn red.
You can also click Wait-for graph to see a simplified process-to-process waiting diagram in the log panel.
STEP 4 Check Banker's Algorithm Tab
Click the Banker's Algorithm tab in the top navigation.

What is Banker's Algorithm?
It is like a bank that only gives loans when it is sure it can still satisfy all customers. The OS only grants a resource request if the resulting state is "safe" — meaning all processes can still eventually finish.

What you will see here:
• Allocation matrix — what resources each process currently holds
• Maximum matrix — the maximum resources each process might ever need
• Need matrix — computed automatically: Need = Maximum − Allocation
• Available — how many of each resource are free right now

If deadlock was detected in Step 3, the result is already shown automatically. Otherwise click Run safety algorithm to check the manually entered values. The system will show whether a safe sequence exists or confirm an unsafe state. Below the result you will also see the RAG snapshot.
STEP 5 Apply Recovery (If Deadlock Was Found)
Click the Recovery tab. If deadlock was detected you will see the victim selection and three strategy cards.

What is a "victim"?
The OS must choose a process to sacrifice to break the deadlock. It picks the one with the lowest cost score: score = (resources held × 2) + pending requests − priority

Choose one of the 3 strategies:
🔴 Process Termination — Kill the victim completely. All its resources are freed instantly. Best when you want the fastest fix. The process loses all progress.
🟡 Resource Preemption — Forcibly take one resource from the victim. Victim is paused and put back in the waiting queue for that resource. Good for when the process has done a lot of work.
🟢 Process Rollback — Roll the victim back to a saved checkpoint. All resources since that checkpoint are released. Least data loss but complex to implement.
After clicking Apply Selected Strategy, the system will:
1. Show the step-by-step recovery process in the log
2. Display the Before (deadlocked, red) and After (recovered, green) RAG graphs side by side
3. Confirm whether the system is now deadlock-free
STEP 6 Learn More in the Learn Tab
Click the Learn tab for theory cards covering:

• How RAG edges represent system state
• DFS vs Wait-for graph vs Banker's — when to use each
• How victim selection and recovery strategies work

This is useful for understanding the concepts before or after running simulations.
